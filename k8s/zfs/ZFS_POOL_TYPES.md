## 디스크 기반 풀 (Disk-based Pool)

### 설명

물리적 디스크나 파티션을 직접 사용하여 ZFS 풀을 생성하는 방법입니다.

### 생성 방법

```bash
# 전체 디스크 사용
sudo zpool create bonanza /dev/sdb

# 파티션 사용
sudo zpool create bonanza /dev/sdb1

# 여러 디스크를 스트라이프로 구성
sudo zpool create bonanza /dev/sdb /dev/sdc

# 미러 구성 (RAID 1)
sudo zpool create bonanza mirror /dev/sdb /dev/sdc

# RAIDZ 구성 (RAID 5)
sudo zpool create bonanza raidz /dev/sdb /dev/sdc /dev/sdd
```

### 특징

- **직접 디스크 접근**: 파일 시스템 계층 없이 디스크에 직접 접근
- **최고 성능**: 오버헤드가 최소화되어 가장 빠른 성능 제공
- **데이터 무결성**: ZFS의 자체 체크섬 및 복구 기능 활용 가능
- **하드웨어 제어**: 디스크 레벨에서의 최적화 가능

### 장점

✅ **최고 성능**: 파일 시스템 오버헤드 없음  
✅ **안정성**: 물리적 디스크 직접 관리  
✅ **확장성**: 실제 디스크 용량만큼 사용 가능  
✅ **데이터 보호**: ZFS의 모든 기능 (스냅샷, 복제, 압축 등) 활용  
✅ **프로덕션 적합**: 실제 운영 환경에 적합  

### 단점

❌ **디스크 필요**: 별도의 물리적 디스크 또는 파티션 필요  
❌ **데이터 손실 위험**: 디스크 포맷 시 기존 데이터 삭제  
❌ **유연성 부족**: 디스크 크기 변경이 어려움

## 파일 기반 풀 (File-based Pool)

### 설명

파일을 루프 디바이스로 마운트하여 ZFS 풀을 생성하는 방법입니다.

### 생성 방법

```bash
# 1. 디렉토리 생성
sudo mkdir -p /var/lib/zfs-pool

# 2. 파일 생성 (예: 200GB)
sudo truncate -s 200G /var/lib/zfs-pool/zfs-pool.img

# 3. 루프 디바이스 생성
sudo losetup /dev/loop0 /var/lib/zfs-pool/zfs-pool.img

# 4. ZFS 풀 생성
sudo zpool create bonanza /dev/loop0

# 또는 직접 파일 사용 (최신 ZFS 버전)
sudo zpool create bonanza /var/lib/zfs-pool/zfs-pool.img
```

### 특징

- **파일 시스템 위에 파일**: 기존 파일 시스템(ext4, xfs 등) 위에 파일로 생성
- **유연성**: 파일 크기 조정이 상대적으로 쉬움
- **테스트 용이**: 실제 디스크 없이도 테스트 가능
- **이중 파일 시스템**: 호스트 파일 시스템 + ZFS 파일 시스템

### 장점

✅ **유연성**: 파일 크기 조정 가능 (일부 제한)  
✅ **테스트 용이**: 실제 디스크 없이도 테스트 가능  
✅ **데이터 손실 위험 낮음**: 기존 디스크 포맷 불필요  
✅ **빠른 설정**: 디스크 파티셔닝 불필요  

### 단점

❌ **성능 저하**: 파일 시스템 오버헤드로 인한 성능 저하  
❌ **이중 오버헤드**: 호스트 파일 시스템 + ZFS 파일 시스템  
❌ **안정성**: 호스트 파일 시스템에 의존  
❌ **용량 제한**: 호스트 파일 시스템의 사용 가능 공간에 제한  
❌ **프로덕션 부적합**: 운영 환경에서는 권장하지 않음

## 성능 차이

### 디스크 기반 풀

```
애플리케이션
    ↓
ZFS 파일 시스템
    ↓
블록 디바이스 (/dev/sdb)
    ↓
물리적 디스크
```

**특징:**
- 단일 파일 시스템 계층
- 직접 블록 디바이스 접근
- 최소 오버헤드
- 높은 I/O 성능

### 파일 기반 풀

```
애플리케이션
    ↓
ZFS 파일 시스템
    ↓
루프 디바이스 (/dev/loop0)
    ↓
파일 (/var/lib/zfs-pool/zfs-pool.img)
    ↓
호스트 파일 시스템 (ext4/xfs)
    ↓
블록 디바이스 (/dev/sda)
    ↓
물리적 디스크
```

**특징:**
- 이중 파일 시스템 계층
- 파일을 통한 간접 접근
- 추가 오버헤드
- 성능 저하 (약 10-30%)

### 성능 벤치마크 예시

```
디스크 기반:
- 순차 읽기: ~500 MB/s
- 순차 쓰기: ~400 MB/s
- 랜덤 읽기: ~50,000 IOPS
- 랜덤 쓰기: ~30,000 IOPS

파일 기반:
- 순차 읽기: ~350 MB/s (30% 저하)
- 순차 쓰기: ~280 MB/s (30% 저하)
- 랜덤 읽기: ~35,000 IOPS (30% 저하)
- 랜덤 쓰기: ~20,000 IOPS (33% 저하)
```

*실제 성능은 디스크, 파일 시스템, 시스템 구성에 따라 다를 수 있습니다.*

## 사용 사례

### 디스크 기반 풀을 사용해야 하는 경우

✅ **프로덕션 환경**: 실제 운영 서버  
✅ **고성능 요구**: 높은 I/O 성능이 필요한 경우  
✅ **대용량 데이터**: 수백 GB 이상의 데이터 저장  
✅ **데이터베이스**: QuestDB, PostgreSQL 등 데이터베이스  
✅ **장기 운영**: 안정적인 장기 운영이 필요한 경우  
✅ **데이터 보호**: 중요한 데이터 보호가 필요한 경우  

### 파일 기반 풀을 사용해야 하는 경우

✅ **개발/테스트 환경**: 로컬 개발 또는 테스트  
✅ **빠른 프로토타이핑**: 빠른 개념 검증  
✅ **학습 목적**: ZFS 기능 학습 및 실험  
✅ **디스크 부족**: 별도 디스크가 없는 경우  
✅ **임시 사용**: 일시적인 스토리지 필요  
✅ **VM 환경**: 가상 머신 내에서 테스트

## 권장사항

### 프로덕션 환경

**✅ 디스크 기반 풀 사용을 강력히 권장합니다.**

이유:
- 최고 성능 제공
- 안정성 및 데이터 무결성
- ZFS의 모든 기능 활용 가능
- 운영 환경에 적합

### 개발/테스트 환경

**파일 기반 풀도 사용 가능하지만, 가능하면 디스크 기반을 권장합니다.**

이유:
- 프로덕션과 유사한 환경 구성
- 성능 테스트 정확도 향상
- 문제 조기 발견

### 하이브리드 접근

1. **개발 단계**: 파일 기반 풀로 빠른 테스트
2. **스테이징 단계**: 디스크 기반 풀로 성능 검증
3. **프로덕션**: 디스크 기반 풀로 안정적 운영

```bash
# 파일 기반 풀 생성 (테스트용)
sudo mkdir -p /var/lib/zfs-pool
```

⚠️ **용량 관리**: 호스트 파일 시스템 용량 모니터링 필요

## 마이그레이션

### 파일 기반 → 디스크 기반

```bash
# 1. 데이터 백업
sudo zfs snapshot bonanza@backup
sudo zfs send bonanza@backup | gzip > backup.gz

# 2. 디스크 기반 풀 생성
sudo zpool create bonanza-new /dev/sdb

# 3. 데이터 복원
gunzip -c backup.gz | sudo zfs receive bonanza-new/restored

# 4. 검증 후 교체
```

## 요약

| 구분 | 디스크 기반 | 파일 기반 |
|------|------------|----------|
| **용도** | 프로덕션 | 개발/테스트 |
| **성능** | 최고 | 보통 (30% 저하) |
| **안정성** | 높음 | 중간 |
| **권장 여부** | ✅ 프로덕션 필수 | ⚠️ 테스트만 |

**결론**: 프로덕션 환경에서는 반드시 **디스크 기반 풀**을 사용하고, 개발/테스트 환경에서만 파일 기반 풀을 고려하세요.
